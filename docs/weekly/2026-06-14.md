## 06.14 学

### 微调和rag的对比

微调（Fine-tune）：训练模型，永久地改变模型的权重，贵，不及时。
RAG 临时地改变模型的上下文，便宜，快速。

### 为什么需要重排序？

RAG现在的检索是这样的：
用户问一个问题 → 把问题转成向量 → 在向量库里找"距离最近"的几个chunk → 直接喂给LLM
这个"距离最近"用的是向量相似度，比如余弦相似度。它衡量的是语义上像不像。
     **但"语义像"不等于"对回答这个问题有用"**。
举个例子：用户问"合同违约的赔偿标准是什么"，向量检索可能给你返回5个chunk：

chunk1：讲违约金计算方式 ✅ 真正有用
chunk2：讲违约的定义 — 语义很像，但对回答没啥用
chunk3：讲合同解除条件 — 也沾边，但不是重点
chunk4：讲赔偿的诉讼流程 — 有点用但不核心
chunk5：讲违约责任的免除情形 — 语义也像，但答非所问

向量检索把这5个都给你了，**排序也不一定对**。如果直接拼进Prompt，**LLM拿到的是一堆噪声混着有用信息，回答质量就下降了**。
重排序做的事就是：拿到这5个候选chunk之后，用一个更精准的模型重新打分，**把真正有用的排到前面，把噪声过滤掉**。
这个"更精准的模型"通常是Cross-Encoder，它不是把问题和chunk分别转向量再比距离，而是把"问题+chunk"拼在一起输入模型，让模型直接判断**这个chunk对这个问题的相关程度**。精度更高，但速度慢，所以只用来**对少量候选结果重排，不用来全量检索**。
### 独立编码 联合编码 
Transformer里有个Self-Attention机制，它能让一段文字里的每个词都去关注其他词。**比如"他打了我"，模型读"打"这个词的时候，能同时看到"他"和"我"，知道是谁打谁**。
>`向量检索 `的做法是：用嵌入模型把问题单独处理一遍，得到一个向量。再把chunk单独处理一遍，得到另一个向量。这两次处理是完全独立的，问题在被处理的时候根本不知道chunk长什么样，chunk在被处理的时候也不知道问题是什么。处理完了才放在一起算相似度。
>`Cross-Encoder`的做法是：把问题和chunk拼成一段话，比如拼成"[问题] 违约赔偿怎么算 [chunk] 违约金不得超过实际损失的130%..."，然后一起喂给模型。模型在处理"违约赔偿怎么算"这几个字的时候，通过Attention机制，能同时看到后面chunk里的内容。*反过来也一样，模型处理chunk内容的时候，也能看到前面的问题。这就是"互相看见对方"*。

所以*独立编码"就是分开处理，处理时互不知情。"联合编码"就是拼在一起处理，处理时互相参考*
结果就是：**联合编码的模型能判断"这个chunk对回答这个具体问题有没有用"**，而不只是"这个chunk和问题语义上像不像"。这两件事不一样。

## 6.16 加 Rerank

在 `retriever.py` 里加了 CrossEncoder 重排序。

**改了什么：**
- FAISS 粗召回从 k=5 改成 k=10（多捞一些候选）
- 加 `CrossEncoder("BAAI/bge-reranker-v2-m3")` 做精排
- 10 条候选按分数排序取 top 3，传给 LLM

**为什么向量检索不够：**
- FAISS 用余弦相似度，衡量的是"语义像不像"
- 但"语义像"不等于"对回答有用"——比如"违约定义"和"违约赔偿标准"语义近，但前者不回答问题
- CrossEncoder 把"问题+chunk"拼一起打分，判断的是"这个chunk对回答这个具体问题有没有用"

**两个概念：**
- 独立编码（Bi-Encoder）：问题和chunk分开转向量，互不知情。速度快，适合全量检索
- 联合编码（Cross-Encoder）：问题和chunk拼一起处理，能互相看见对方。精度高但慢，只适合精排少量候选

**用到的新语法：**
- `zip(scores, docs)`：两个列表按位置配对，像拉链
- `for _, doc in pairs`：`_` 表示这个变量不用，占位而已

*tool_calls*里包含：工具名称（name）、工具参数（args）、调用ID（id）。
*AIMessage*里有：
- `content`：模型输出的文字内容
- `tool_calls`：如果模型决定调用工具，就在这里，没有就是空列表

## 6.16 LangGraph 化

把 `create_agent()` 黑盒拆成手写 StateGraph。

**三个核心概念：**

| 概念 | 是什么 | 类比 |
|------|--------|------|
| **State** | 图里流转的数据包，每个节点读写它 | 信封，往里塞东西再传出去 |
| **Node** | 图里的步骤，一个 Python 函数 | 流水线上的工位 |
| **Edge** | 节点之间的箭头 | 流水线传送带 |

两种边：固定边（`add_edge`，永远从 A 到 B）和条件边（`add_conditional_edges`，看判断函数的结果决定走哪）。

**写了什么：**

1. `AgentState` — 声明数据格式，`messages` 字段用 `add_messages` 自动追加新消息
2. `llm.bind_tools(tools)` — 把 @tool 的 docstring 转成 function calling 格式，LLM 才知道有工具
3. `call_model(state)` — 调绑了工具的 LLM，返回 AIMessage。**必须用绑了的 LLM，裸 LLM 不会调工具**
4. `ToolNode(tools)` — LangGraph 内置，自动解析 tool_calls → 执行工具 → 返回 ToolMessage
5. `should_continue(state)` — 看最后一条消息有没有 tool_calls，有 → `"tools"`，无 → `"__end__"`
6. `graph.compile()` — 把图纸编译成能 `.ainvoke()` 的引擎

**关键理解：**
- `"llm"` 和 `"tools"` 只是节点名字标签，给边用的
- 不用工具，`should_continue` 返回 `"__end__"`，不经过 tools 节点
- `add_messages` 是 LangGraph 的消息合并器，节点只管返回新消息，系统自动追加
- SYSTEM_PROMPT 从 `create_agent(system_prompt=...)` 自动注入 变成 手动 `messages.insert(0, SystemMessage(...))`

**新旧对比：**

| | 旧 `create_agent()` | 新手写 StateGraph |
|---|---|---|
| 代码量 | 一行 | ~30 行 |
| 能否定制 | 黑盒 | 每个节点/边都是自己的 |
| SYSTEM_PROMPT | 自动注入 | 手动插入 |
