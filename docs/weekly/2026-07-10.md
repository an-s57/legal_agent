### llm随机性

llm生成文字的过程，本质就是从候选词里面随机挑选。llm有一个temperature参数，temperature=0，确定性。
temperature = 0：永远选概率最高的（确定性）

temperature = 1（默认）：按概率分布随机选，60% 的时候选第一个，35% 的时候选第二个

temperature = 2：更随机，低概率的也有可能被选中

GLM-4-flash 默认 temperature 不是 0，所以同样的问题，今天可能选 RAG，明天可能选联网。

怎么解决？
设 temperature = 0（但智谱 API 不一定支持）

### state的一些回忆
LangGraph 的设计规则：每个节点函数只能**通过 state 参数拿到数据，不能直接访问外部变量**。

def call_planner(state: AgentState):
    case_summary = state["case_summary"]    # 从 state 里取
    user_message = state["messages"][-1]    # 从 state 里取

因为 LangGraph 是一个**流水线框架**，它需要**管理数据在节点之间的流动**。如果每个节点都直接读外部变量，LangGraph 就不知道谁依赖谁，没法控制流程。所有数据都走 state，LangGraph 就能清楚地看到。

state = {
    **"messages"**: [
        SystemMessage(content="你是一个法律助手..."),
        HumanMessage(content="商家跑路了怎么办"),
    ],
    **"case_summary"**: '{"case_type":"消费纠纷","event_description":"商家跑路"}',
    **"info_complete"**: False,   # planner 节点执行后才有这个值
}

每个节点可以**读state里面的节点，也可以返回修改后的字段**.

### 0716 学
#### 2层记忆机制
两层记忆解决的是上下文长度和有效信息密度的矛盾。sessions[session_id] = {"history": [], "case_summary": {}}

短期记忆（history）：**保存完整的对话历史——包括 HumanMessage、AIMessage、ToolMessage**。*为 ReAct 循环提供前置知识*，**否则react循环没有办法进入有效的推理和检索**，LLM 需要看到完整的工具调用链路才能做出正确决策。

长期记忆（case_summary）：用 LLM 把对话中的案情信息*增量提取为结构化 JSON*——案件类型、事件描述、发生时间、损失、用户诉求五个字段。**每轮对话结束后更新，注入下一轮的 SystemMessage。**

为什么要两层？因为用户可能聊几十轮，history 越来越长，LLM 会遗忘早期的案情细节。case_summary 像一个"压缩备忘录"，无论聊多久，关键事实永远在 system prompt 里，不会丢。同时 history **保留完整上下文，保证对话连贯性**。

我的项目有双层记忆机制。短期记忆**存原始对话历史，保证上下文连贯**；长期记忆用 **LLM 增量提取结构化案情摘要**（案件类型、事件、时间、损失、诉求），**注入下一轮的 SystemMessage**。这样做的核心价值有两个：**一是 Token 消耗恒定，案情摘要只有几百字，不会像原始历史那样线性增长**；二**是让 ReAct 循环更高效——Agent 不用从冗长的对话历史里自己提炼案件背景**，直接拿到结构化信息就能快速决策该检索什么法条，此外**case_summary让PLAANER节点**不用重读完整对话历史，就可以判断对话是否完整，这就是**压缩上下文**的价值。


#### rerank
FAISS 做的是**双塔（bi-encoder）检索**：query 和文档**分别用** embedding 模型编码成向量，然后算*余弦相似度*。快，但它是"各自编码再比对"，缺乏 query 和文档之间词粒度的交互，容易召回**语义相近但不真正相关的内容**。

Reranker 用的是 **CrossEncoder（交叉编码器）**：*把 query 和文档拼成一对一起输入模型*，模型逐词交叉注意，*判断这对 query-doc 的相关性得分*。精度远高于向量相似度，但速度慢（不能对全库跑）。

所以我的策略是**粗排+精排两阶段：FAISS 粗召回 20 条（快），Reranker 对 20 条重新打分取 Top 5（准）**。用 CrossEncoder 只跑 20 条，本地 CPU 完全够用，不额外花钱。

模型用的是 BAAI 的 **bge-reranker-base**，*中文法律文本效果很好*。

#### 整个agent的流程

整个流程是这样的：

1.用户发消息

2、FastAPI 的 POST /legal/chat/stream 接收请求

3.*加载会话上下文* 从内存中取出这个 session 的 history 和 case_summary

4.Planner 节点 ，用 LLM 分析用户输入，检查5个维度是否齐全：案件类型，事件描述、发生时间、损失/后果、用户诉求。缺了就生成追问直接返回给用户，不进检索；全了就放行

5.LLM 节点（ReAct 循环入口），LLM 拿到 **history + case_summary + 用户消息**，推理是否需要调用工具。**如果需要，输出 tool_calls；如果不需要，直接生成回答**

6.Tools 节点 → 执行工具（RAG 检索或联网搜索），**把工具返回的结果封装成 ToolMessage**，写回对话历史

7.条件跳转 → **有 tool_calls 就回到 LLM 节点（带着工具结果再推理）**，没有就结束循环，输出最终回答

8.SSE 流式输出 → 回答逐 token 推送给前端

9.更新记忆 → 新的一轮对话追加到 history，同时 LLM -**增量更新** case_summary

tool_calls 是 LLM 节点的输出，是一个列表，每个元素包含：

字段	含义	示例
`name`	要调哪个工具	"legal_rag_search"
`args`	传给工具的参数	{"query": "拖欠工资劳动仲裁时效"}
`id`	唯一标识	"call_abc123"

→ 意思是 LLM 说："我需要查法条，帮我调 legal_rag_search，搜这个关键词。"

ToolMessage 是 Tools 节点执行工具后返回的结果，每个包含：

字段	含义	示例
`content`	工具返回的实际内容	检索到的 5 条法条原文
`tool_call_id`	*对应哪个 tool_call	"call_abc123"*
`name`	工具名称	"legal_rag_search"

→ 意思是"这是你刚才让我查的结果，拿去看。"

**LLM 输出 tool_calls → Tools 节点执行 → 返回 ToolMessage → LLM 看到 ToolMessage 再推理**

#### 为什么用langgraph手写stategraph

create_agent() 是 LangChain 封装的高级 API，一行代码就能创建 Agent。**但它的问题在于黑盒——状态结构、节点流转、工具绑定全部封装好了，改不了。**

我手写 StateGraph 原因：

**流程可控**：我想在 ReAct 之前插入 Planner 节点做信息完整性检查，黑盒方案做不到。手写的话，新增一个节点函数再加一条边就行，不影响原有逻辑

**状态透明**：我自定义了 AgentState，里面有哪些字段（messages、case_summary、tools_used）我自己清楚。每个节点的输入输出都可以打日志，调试时一目了然



#### 你为什么不直接用智谱的 Embedding API，而是用 Ollama 本地跑 nomic-embed-text
1.成本，这个模型它需要计费但是本地的online完全免费

2.隐私，如果用智谱的话就需要经过外部API,我们这个可以在本地跑，离线也可以用

当然也付出了一定的代价，就是这个模型**精度肯定是不如智谱的embedding模型**,但是我们后面的重排序弥补

#### ollama版本问题
LangChain 官方的 langchain_ollama.OllamaEmbeddings 跟**我本机装的 Ollama 客户端版本不兼容**——新版 Ollama 的 API 路径变了，但 LangChain 那个封装还是调旧接口，直接报错。

所以我自定义了一个类，核心就两件事：

直接用 httpx **发 HTTP 请求调 Ollama 的 /api/embeddings 接口**，不经过 LangChain 那一层封装，绕开了版本兼容问题
设置 trust_env=False，**防止请求被系统代理拦截**——Windows 上经常配了代理，httpx 默认会读系统代理设置，导致发向 127.0.0.1:11434 的请求也被代理，连接失败。关掉就好了
本质上就是一个轻量 HTTP 封装，十几行代码。遇到库版本不兼容的时候，直接调底层 API 往往是最快的解法。
