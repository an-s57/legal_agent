### llm随机性

llm生成文字的过程，本质就是从候选词里面随机挑选。llm有一个temperature参数，temperature=0，确定性。
temperature = 0：永远选概率最高的（确定性）

temperature = 1（默认）：按概率分布随机选，60% 的时候选第一个，35% 的时候选第二个

temperature = 2：更随机，低概率的也有可能被选中

GLM-4-flash 默认 temperature 不是 0，所以同样的问题，今天可能选 RAG，明天可能选联网。

怎么解决？
设 temperature = 0（但智谱 API 不一定支持）

### state的一些回忆
LangGraph 的设计规则：每个节点函数只能**通过 state 参数拿到数据，不能直接访问外部变量**。

def call_planner(state: AgentState):
    case_summary = state["case_summary"]    # 从 state 里取
    user_message = state["messages"][-1]    # 从 state 里取

因为 LangGraph 是一个**流水线框架**，它需要**管理数据在节点之间的流动**。如果每个节点都直接读外部变量，LangGraph 就不知道谁依赖谁，没法控制流程。所有数据都走 state，LangGraph 就能清楚地看到。

state = {
    **"messages"**: [
        SystemMessage(content="你是一个法律助手..."),
        HumanMessage(content="商家跑路了怎么办"),
    ],
    **"case_summary"**: '{"case_type":"消费纠纷","event_description":"商家跑路"}',
    **"info_complete"**: False,   # planner 节点执行后才有这个值
}

每个节点可以**读state里面的节点，也可以返回修改后的字段**.

## 0714 续 — 评测修复 + Planner 设计 + 模型选型

### 评测系统的逻辑错误：error 被当成"答错"

之前在 `evaluate_tool_selection` 里，如果一道题超时或报错，`correct` 字段是 `False`，统计时被算成"答错"。

比如10道题里2道超时，8道答对：
- 之前算出来：`8/10 = 80%`
- 实际应该是：`8/8 = 100%`，那2道是服务器问题，不是 Agent 的锅

**修复方案**：把每道题的 `round_results` 按状态分类

| 情况 | status | 怎么处理 |
|------|--------|---------|
| 全部 round 成功 | `"ok"` | 正常参与统计 |
| 部分 round 报错 | `"partial"` | 用成功的 round 投票 |
| 全部 round 报错 | `"error"` | **排除在准确率之外**，单独报告 |

输出变成：
```
工具选择准确率: 8/8 = 100.0%
  ⚠️  2 题因超时/报错被排除，不计入准确率
```

**学到的东西**：评测系统本身的逻辑错误比被测系统的 bug 更难发现。数据污染比代码 bug 更隐蔽——代码会报错，数据污染不会报错，只会给你一个"看起来合理"的错误数字。

### Planner 追问：要不要调两次 LLM？

**方案 A**：Planner 分析缺什么 → 用模板拼接追问文本
**方案 B**：Planner 一次 LLM 调用，同时输出 `missing_fields` + `follow_up_question`

选了 B。但关键洞察是：**方案 B 并不比方案 A 多花钱**。

因为 Planner "分析缺什么信息"这步本身就是 LLM 调用——你不可能用 if-else 判断用户的咨询缺不缺法律要素。所以真正的选择是：

| | 分析 | 生成追问 | 总调用 |
|---|---|---|---|
| A | LLM ×1 | 字符串拼接 | **1 次** |
| B | ——— LLM ×1 顺便生成追问 ——— | **1 次** |

同样的调用次数，B 的追问更自然，没有理由选 A。

### 模型选型：GLM-4.7 替换 GLM-4-flash

手里有 500 万 GLM-4.7 tokens，对比了一下：

| | GLM-4-flash（当前） | GLM-4.7 |
|---|---|---|
| 参数 | 30B / 3B 激活 | ~400B / 20B+ 激活 |
| SWE-bench | 59.2% | **73.8%** |
| HLE | 14.4% | **42.8%** |
| 价格 | 免费 | 输入 ¥4.7/M tokens |

跨了两个数量级的差距。工具选择不稳的问题，换 GLM-4.7 应该直接好一截。决定把 `legal_agent.py` 里的模型换成 `glm-4.7`。

顺便对比了 GLM-4.7 和 DeepSeek V4 Pro（就是 Claude Code 背后跑的那个）：
- Code Arena：DS #14 vs GLM #15，只差 16 ELO，基本忽略
- 中文：GLM-4.7 更好
- 价格：GLM 输入更便宜，DS 输出更便宜

**结论**：两个同梯队，中文法律场景 GLM-4.7 更合适。
