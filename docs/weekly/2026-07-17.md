###  介绍项目
我这个项目它解决的问题就是当普通用户面对法律问题时怎样去描述案情查找相关的法律根据还有采取下一步行动的问题。

然后然后如果信息完整后我们进入LLM节点去根据用户的提问来判断使用什么工具是选择联网搜索还是本地法律文档库检索。
最后整合答案通过SSE流式输出。此外呢我还进行了多轮历史记录和案情结构化摘要。
而且我还对检索呢进行了一个测评，目前是17道题目中有14道命中了们的这个需要。

### 遇到的最大问题

我遇到的核心难点是：**如何让 RAG 评测结果可复现、可信**

参考回答：
项目中最难的部分是**建立可信的 RAG 评测**。初期我发现同类测试的*命中率波动很大*，一开始误以为是电脑性能或模型下载不完整导致的。后来我意识到，RAG 的结果不只是**向量相似度，还受向量库版本、文档切分、召回数量、Reranker 是否成功加载以及异常降级策略影响**。  
我目前先统一了线上和评测的 k、top_k 参数。下一步会为每次检索记录模型版本、向量库版本、检索参数和是否发生 fallback；如果 Reranker 失败，**产品仍然返回 FAISS 结果保证可用性**，但评测会将该次**标为降级结果**，不和正常 Reranker 结果混在一起计算。这个过程让我认识到，工程上不仅要让功能跑通，还要保证指标可解释、实验可复现。

于是我就又要开始修改了，，


模型信息：模型名、版本、是否加载成功、运行设备。  
向量库信息：法律文档集合、切分参数、Embedding 模型、索引版本。  
检索链路信息：k、top_k、Reranker 是否成功、是否 fallback、耗时。

#### 如果 Reranker、Ollama 或大模型 API 挂了，系统会怎样？

当前实现：

Reranker 失败：会回退到 FAISS 原始结果。这是当前唯一明确做了**降级处理**的组件。
Ollama Embedding 服务失败：legal_rag_search 当前没有完整的异常兜底，检索工具很可能失败，进而影响整次请求。
GLM API 失败：当前也没有统一的超时、重试、SSE 错误事件处理，请求可能直接失败。
DuckDuckGo 搜索失败：这个其实你已经处理了，web_legal_search 的 except 会返回“搜索失败：…”的文本。

未来上线改进：

Ollama 不可用：明确提示**法律检索服务暂不可用，请稍后重试**，不要让模型编造。
GLM API 不可用：前端显示**请求失败和重试入口**；后端加超时、有限重试、错误日志。
Reranker 不可用：继续 FAISS 降级，但记录降级次数。
联网搜索失败：**提示“未获取到最新外部信息”**，可继续用本地库回答，但不能声称信息最新。
你现在可以这样答：
当前我只对 Reranker 做了 FAISS 降级处理。Ollama 和 GLM API 的失败处理还不完善，这是我后续优先补的工程能力；DuckDuckGo 搜索已有异常提示。上线时我会把不同组件的失败状态明确展示给用户，而不是让系统假装正常。


#### 会话数据放内存有什么问题？如果上线怎么改？

sessions = {
  "session-001": {
    "history": [...],
    "case_summary": {...}
  }
}
这个字典在 Python 程序的内存里。
“后端历史会丢失”的意思是：
你关闭服务；
或终端里重新运行 python main.py；
Python 进程重新启动；
内存被清空；
sessions 又变成空字典。
之前的对话和案情摘要就没了。

当前使用**内存字典**，*服务重启会丢失数据*，也不适合多实例部署。后续会通过**用户认证获得 user_id**，并将*会话和案情摘要*存入持久化数据库；Redis 可以用于短期缓存，但数据库才是长期数据来源。


#### 为什么要用 SSE 流式输出？

LLM**调用工具和生成回答的时间耗时不稳定**。SSE**允许服务端生成一部分内容后**立刻推送给前端，用户不用等待整段答案都生成后才能得到反馈。用于**改善首段响应时间和等待体验**
#### 法律助手答错、引用过期法律、泄露用户案情，怎么处理？

第一，法律来源优先使用**权威网站和本地确认过的法律文本**，并**展示来源、条款和生效时间。**
第二，*证据不足或检索失败*时，明确**提示用户核实，而不是给出确定结论**。
第三，**构建评测集**，检查引用是否正确、是否使用过期法律、*高风险问题是否被正确拒答或提示人工咨询*。
第四，隐私方面要最小化收集用户案情，**避免在日志中保存完整敏感信息**，并*评估第三方模型 API 的数据处理条款*；本地模型是一种选择，但不是唯一方案。
第五，涉及重大财产、人身或刑事风险时，**建议用户咨询专业律师**。
#### 你觉得项目最大不足是什么？下一步怎么改？

我认为最大不足是**评测只集中在检索来源命中**，**不能充分证明最终回答的准确性、引用完整性和法律时效性**。  
下一步我会先建立端到端评测：固定问题集、期望来源、引用检查和人工抽样；再补用户会话持久化、错误处理和权威法律来源。多 Agent 审核是后续可探索的方案，但会增加成本和延迟，不是当前第一优先级。
